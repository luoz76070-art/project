# 阶段一轻量 URL -> Markdown -> CSV 优化设计

## 一、背景

当前阶段一 `@zhaoping/tencent-docs-recruiting-extractor` 的主要输入仍以腾讯文档提取流程为中心，核心边界是：

- 从腾讯文档源表读取数据
- 标准化为 `recruiting.csv`
- 将该 CSV 作为阶段二输入

这个设计在腾讯文档单一来源场景下成立，但已经无法覆盖新的业务需求：

- 阶段一输入不一定是腾讯文档配置
- 未来可能需要直接处理学校招聘网站 URL
- 这些 URL 对应的页面不一定是结构化表格，可能只是公告正文页
- 未来还可能出现网页正文 + 附件并存的采集场景

在这种背景下，阶段一最需要解决的问题不是“做一个大而全的多源采集框架”，而是先把当前系统从“腾讯文档专用输入”扩展成“网页正文型 URL -> Markdown -> CSV”的通用入口，并继续兼容阶段二。

## 二、目标

本次优化目标聚焦在一个轻量可落地闭环：

1. 阶段一新增网页正文型 URL 输入能力
2. 使用 `url-to-markdown` 能力把网页正文型 URL 页面转成 markdown
3. 从 markdown 中提取招聘字段
4. 将提取结果转换成当前阶段二可直接消费的 CSV 结构
5. 保持阶段二和阶段三输入边界不变，不要求同步重构后续阶段

## 三、非目标

本次不做以下事项：

- 不建设完整的多源采集框架
- 不在本次引入数据库
- 不开发前端或后端 API
- 不一次性支持所有学校网站模板
- 不把附件解析、图片 OCR、复杂 PDF 识别全部做完
- 不改变阶段二和阶段三当前的主输入契约

## 四、推荐路线

本次采用轻量路线：

```text
网页正文型 URL -> Markdown -> 招聘字段提取 -> 当前兼容 CSV
```

而不是先做完整的候选字段、评分、复杂状态机体系。

选择原因：

- 目标明确：扩展阶段一的网页正文型 URL 输入能力
- 与当前阶段二最兼容
- 可以先快速支持学校招聘公告类网页
- 后续若需要更复杂的结构，再在此基础上演进

## 五、总体架构

阶段一本次优化后的轻量结构拆成三个最小单元：

1. `fetchUrlAsMarkdown(url)`
2. `extractRecruitingFields(markdown)`
3. `normalizeFieldsToCsvRow(fields)`

它们共同构成一个新的 URL 采集路径，但不推翻现有腾讯文档路径。

### 5.1 URL 抓取层

负责接收 URL，并调用 `url-to-markdown` 能力输出 markdown 内容。

该层输出至少应包含：

- `sourceUrl`
- `title`
- `markdownContent`
- `htmlSnapshotPath`（若可用）
- `fetchedAt`

这一层只解决“页面怎么拿到”，不负责招聘语义理解。

### 5.2 字段提取层

负责从 markdown 中提取招聘相关字段。

第一版不追求全站点泛化，而是聚焦于公告正文常见字段：

- 公司名称
- 批次
- 企业性质
- 行业大类
- 招聘对象
- 招聘岗位
- 工作地点
- 更新时间
- 截止时间
- 官方公告文字
- 官方公告链接
- 投递方式
- 投递链接
- 备注

如果部分字段无法提取：

- 允许输出空值
- 但必须保留来源 URL 和原始 markdown
- 不要求在本次设计中引入复杂候选字段评分体系

### 5.3 标准化输出层

负责把提取结果投影成当前阶段二可消费的 CSV 行。

也就是说，输出仍然兼容当前 `recruiting.csv` 及阶段二依赖的标准字段结构，不额外发明新的下游契约。

## 六、与 `url-to-markdown` 的关系

### 6.1 为什么使用它

`url-to-markdown` 已经提供了：

- 浏览器渲染抓取能力
- 将 URL 页面内容转换为 markdown 的能力
- HTML snapshot 保留能力

这些能力非常适合阶段一的“网页正文型 URL”输入扩展。

### 6.2 如何接入

不建议让阶段一业务逻辑直接依赖 skill 的所有内部细节，而建议：

- 先把它当作 fetch 能力来源
- 在项目内部包一层轻量适配函数
- 由阶段一的 URL 路径统一消费该适配层输出

### 6.3 后续演进空间

如果未来 `url-to-markdown` 依赖逐渐稳定，可以：

- 继续外部调用 skill
- 或把其中成熟脚本逐步内化到项目内部

但这不是本次目标。

## 七、输入与输出设计

### 7.1 新增轻量输入模型

建议新增一个简单 URL 输入模型：

```js
{
  url: string,
  snapshotDate?: string
}
```

当前不需要过早扩展为复杂 source type 系统。

### 7.2 Markdown 中间结果

建议阶段一内部保留一个轻量中间结构：

```js
{
  sourceUrl: string,
  title?: string,
  markdownContent: string,
  htmlSnapshotPath?: string,
  fetchedAt: string
}
```

### 7.3 最终兼容输出

最终输出仍然是当前阶段二兼容的 CSV 记录和 summary 信息。

也就是说：

- 阶段二不需要因为这次优化立即重写
- 新增 URL 路径只是在阶段一内部增加新的“来源到标准记录”的生成方式

## 八、错误处理策略

本次轻量路线下，错误处理保持简单：

1. URL 抓取失败
   - 记录失败 summary
   - 不生成有效 CSV

2. markdown 生成成功，但字段提取失败
   - 保留 markdown / 快照
   - 在 summary 中记录“提取不足”
   - 直接判定为失败，不生成阶段二兼容 CSV，但必须明确记录原因

3. 字段不完整
   - 当前版本允许字段留空
   - 但必须至少保留来源 URL 和必要上下文

复杂候选字段评分与置信度机制留待后续版本。

## 九、与当前三阶段的关系

### 9.1 阶段一

- 增加新的 URL 路径
- 保留原腾讯文档路径

### 9.2 阶段二

- 不改输入契约
- 继续消费阶段一标准 CSV 输出

### 9.3 阶段三

- 不受影响
- 继续消费阶段二产物

## 十、范围控制

为了保证这次优化可落地，本次建议只先支持：

- 腾讯文档原路径继续保留
- 网页正文型招聘 URL 新路径

不把以下内容纳入第一版：

- 页面中的附件下载解析
- PDF/Excel 附件结构化
- 图片 OCR
- 多站点模板泛化评分体系

## 十一、推荐实施顺序

1. 新增 URL -> markdown 抓取适配层
2. 新增 markdown -> 招聘字段提取逻辑
3. 新增字段 -> 当前 CSV 行标准化逻辑
4. 补测试，确认阶段二仍能消费输出
5. 保留原腾讯文档路径并行存在

## 十二、结论

本次阶段一优化的推荐方向不是“完整多源采集平台”，而是：

**先把阶段一升级为 URL -> Markdown -> CSV 的通用采集入口。**

这样能用最小的改动，扩展学校招聘网站等网页正文型来源，同时继续兼容当前阶段二与阶段三。
