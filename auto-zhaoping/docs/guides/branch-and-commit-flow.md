# 分支与提交流程

这份文档放在 `docs/guides/branch-and-commit-flow.md`，用于把一次本地改动如何变成 GitHub 上的 PR 讲清楚。

## 1. 整体流程

```text
新建分支 -> 修改文件 -> git add -> git commit -> git push -> 创建 PR -> 审查 -> 合并
```

## 2. 每一步在做什么

### 新建分支

- 用分支隔离当前任务。
- 这样不会直接影响 `main`。

### 修改文件

- 在本地完成代码、文档或配置改动。
- 可以先改一部分，再继续补完。

### `git add`

- 把你想提交的改动放进暂存区。
- 最常见的写法是 `git add .`，但也可以只加某几个文件。

### `git commit`

- 把暂存区内容保存成一个提交。
- 提交信息最好简短、具体。

### `git push`

- 把本地分支推到 GitHub。
- 这样其他人或你自己就能在网页上看到这次改动。

### 创建 PR

- PR 是“请求把分支改动合并到主分支”。
- 它是 GitHub 上最常见的协作入口。

### 审查与合并

- 在 PR 页面检查改动、测试结果和讨论。
- 没问题后再合并进 `main`。

## 3. 常见命令

```bash
git status
git diff
git add .
git commit -m "message"
git push -u origin <branch-name>
```

## 4. 什么时候要先看 `git diff`

- 提交前想确认改动是否正确。
- 想避免把不相关文件一起提交进去。
- 想检查文档或日志是否写得准确。

## 5. 什么时候要先看 `git status`

- 不确定当前有哪些改动。
- 不确定有没有未跟踪文件。
- 不确定现在是不是在正确分支上。

## 6. 这次仓库里的典型场景

- 我们通常先在分支上整理某个阶段。
- 测试通过后再推送到 GitHub。
- 然后通过 PR 让改动进入主分支。

## 7. 最短记忆法

- `git status`：看当前状态
- `git diff`：看改了什么
- `git commit`：保存一个版本点
- `git push`：推到 GitHub
- PR：申请合并到主线
