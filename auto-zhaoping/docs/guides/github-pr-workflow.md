# GitHub PR 入门

这份文档放在 `docs/guides/github-pr-workflow.md`，用于说明 GitHub 上的分支、`main` 和 PR 是怎么配合工作的。

## 这三个概念是什么

### `main`

- 仓库的主分支。
- 通常代表已经确认可用、对外默认的正式版本。
- 合并到 `main` 的内容，一般就是最终要保留的代码或文档。

### 分支

- 你在本地单独做修改时，通常先放在一个分支里。
- 分支的作用是隔离工作，避免直接影响主线。
- 你可以把它理解成“临时工作区”。

### PR

- PR 是 Pull Request，也叫“合并请求”。
- 它表示“我已经在某个分支上改好了，现在请求把这些改动合到主分支”。
- PR 页面会展示改动内容、测试状态和讨论记录。

## 这次任务里发生了什么

1. 我先在本地分支里完成文档整理。
2. 然后把这个分支推到 GitHub。
3. 再在 GitHub 上创建 PR。
4. 这样你就可以在网页上查看全部改动，并决定是否合并。

## 你平时怎么用它

最常见的流程是：

1. 新建一个分支。
2. 在分支上修改文件。
3. 提交 commit。
4. `git push` 到 GitHub。
5. 创建 PR。
6. 在 PR 页面检查改动和测试结果。
7. 没问题后把 PR 合并进 `main`。

## PR 页面里常见的内容

- `Files changed`：这次改了哪些文件。
- `Conversation`：讨论区。
- `Checks`：测试是否通过。
- `Merge pull request`：把 PR 合并进主分支的按钮。

## 什么时候你需要关心它

- 想把本地改动交给别人审查时。
- 想把一批改动正式合并到主线时。
- 想确认测试有没有过、改动是否安全时。

## 最短记忆法

- 分支 = 临时工作区
- PR = 提交审核
- `main` = 正式版本

## 在这个仓库里怎么理解

- `main` 保存的是仓库当前认可的正式状态。
- 你在分支上完成整理、修复或功能开发。
- PR 用来把分支里的成果并入 `main`。

如果你以后还想看“这个仓库里从本地改动到 GitHub 合并”的完整流程，就从这份文档开始看。
